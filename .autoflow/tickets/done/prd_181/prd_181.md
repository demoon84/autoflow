## Core Scope

- Goal: 현재 worker token usage가 86,004,270,902로 집계되는 원인을 `.autoflow/telemetry/runs.jsonl`의 비정상 token row와 `telemetry-project.sh token-usage` 합산/필터링 경로에서 좁혀 수정하고, budget preflight가 불가능한 telemetry 값을 hard block으로 소비하지 않도록 한다.
- In Scope:
  - `packages/cli/telemetry-project.sh token-usage`가 `--since` / `--until` 필터를 유지하면서 `token_input` + `token_output`만 합산하는지 확인하고, 숫자 변환/단위/row validation을 보강한다.
  - `.autoflow/telemetry/runs.jsonl`에 존재하는 비현실적 row 예시(`Todo-176`, `token_input=43000004027`, `token_output=43000000020`)처럼 단일 호출 물리 한계를 크게 넘는 값은 합산에서 제외하거나 별도 warning evidence로 분리한다. 기존 운영 telemetry 파일은 삭제하거나 직접 rewrite하지 않는다.
  - `packages/cli/run-role.sh`와 `runtime/board-scripts/run-role.sh`의 token marker 파싱이 board markdown, wiki snippet, fingerprint, stdout byte length를 token count로 오인하지 않도록 명시적인 adapter token marker만 받아들이고, fallback byte estimate는 bytes/4 수준의 bounded estimate로 유지한다.
  - budget preflight는 sanitized `token-usage` 결과만 사용한다. 결과가 quota 대비 명백히 불가능한 경우(`quota * sanity multiplier` 이상 등)는 `token_budget_exceeded` 대신 telemetry sanity warning/skip marker를 남기고 adapter 진행을 hard block하지 않는다.
  - smoke test를 추가해 corrupt telemetry row(수십B token), 정상 fresh row, `--since` window를 같이 넣었을 때 `token-usage`가 corrupt row를 합산하지 않고 정상 단위의 값을 반환하는지 검증한다.
  - `order_167` / `prd_179`의 stale token-cache source freshness guard, `order_168` / `prd_180`의 repeated preflight circuit breaker와 충돌하지 않게 한다. 이번 PRD는 "최신 source가 있어도 row 값 자체가 불가능한 경우"를 다룬다.
- Out of Scope:
  - 과거 `.autoflow/telemetry/runs.jsonl` 운영 데이터 삭제, rewrite, backfill.
  - `prd_179` / `Todo-178`의 stale token-cache freshness/source precedence 전체 재구현.
  - `prd_180` / `Todo-179`의 repeated preflight circuit breaker 전체 재구현.
  - token quota 기본값 또는 비용 환산 UI 재설계.
  - runner 또는 OS process stop/restart/kill, worktree 삭제, `git push`.
