## Core Scope

- Goal: `autoflow skill list` 가 usage.json sidecar 를 매 스킬마다 4번씩 Python 으로 다시 읽는 구조를 1회 일괄 로드로 바꿔 탭 진입 시간을 단축하고, `autoflow skill view` 에 `--no-bump` 옵션을 추가해 데스크톱 `controlSkill({action: "view"})` 가 manual 클릭에서는 `view_count` / `last_viewed_at` 을 갱신하지 않도록 한다. AGENTS.md rule 18e 가 정의한 `view_count` (RAG retrieval 신호) 는 그대로 보존된다.
- In Scope:
  - `packages/cli/skill-project.sh`
    - `run_list` (line ~800) 의 per-skill `usage_get_field` 호출 (`view_count`, `success_count`, `last_used_at`, `last_viewed_at` 4회 × N 스킬) 을 단일 Python 인보케이션으로 교체한다. 새 helper(예: `usage_dump_all`) 가 usage.json 을 한 번만 파싱해 `key→{view_count,success_count,last_used_at,last_viewed_at}` 매핑을 stdout 으로 emit 하고, run_list 가 이를 associative array (bash 4 호환 또는 임시 파일 lookup) 로 받아 enumerate 결과에 join 한다. 출력 key=value 형식과 필드명은 그대로 유지한다 (renderer 파서 호환).
    - `run_view` 가 새 인자 `--no-bump` 를 받으면 `usage_record_event` 호출을 skip 한다. dispatcher `case "view")` 블록에서 ref 파싱 후 trailing 인자로 `--no-bump` 를 인식한다. 출력에는 여전히 현재 `view_count` / `last_viewed_at` 을 표시 (read-only) 하므로 stdout 형식은 변경하지 않는다.
    - 기존 CLI 사용자의 default 동작 (`autoflow skill view <ref>` → bump) 은 그대로 유지한다.
  - `apps/desktop/src/main.js`
    - `controlSkill` 의 `view` 분기에서 `options.bumpUsage === false` 일 때 args 끝에 `--no-bump` 를 추가한다. 다른 분기(`list`, `archive`, `history`) 는 변경하지 않는다.
    - `allowedSkillActions` set 은 그대로 둔다.
  - `apps/desktop/src/preload.js`
    - 기존 generic forward (`(options) => ipcRenderer.invoke("autoflow:controlSkill", options)`) 가 이미 객체 전체를 넘기므로 코드 변경 없음. 변경 시에도 generic 형태 유지.
  - `apps/desktop/src/renderer/vite-env.d.ts`
    - `controlSkill` options 타입에 `bumpUsage?: boolean` 추가. 기존 필드 시그니처 보존.
  - `apps/desktop/src/renderer/main.tsx`
    - `viewSkill` (line ~1718) 의 `controlSkill({action: "view", ...})` 호출에 `bumpUsage: false` 를 전달한다. 호출 후 list 의 `viewCount` 를 `parsed.viewCount` 로 덮어쓰는 기존 라인(line ~1745) 은 그대로 둔다 — read-only 응답에서도 서버가 현재 카운터를 돌려주므로 단조 증가만 깨지지 않으면 동일.
    - `parseSkillViewOutput` 시그니처/필드는 변경하지 않는다.
- Out of Scope:
  - usage.json 의 sidecar 스키마 변경 (atomic write, key 구조, frontmatter 호환 모두 유지).
  - `view_count` / `manual_open_count` 같은 새로운 카운터 추가 (rule 18e 의 단일 RAG-retrieval 신호 정의 유지). 후속 PRD 에서 검토 가능하지만 본 PRD 범위 아님.
  - `autoflow wiki query --rag` 측 view_count 증분 로직 변경.
  - 데스크톱 skills 패널 UI 레이아웃, 정렬, 필터, history 표시 변경.
  - `autoflow skill list` 의 enumerate 단계 (`enumerate_skill_entries`) 자체 변경. usage join 구간만 최적화한다.
  - `autoflow skill match` / `validate` / `archive` / `create` 의 동작 변경.
  - 새 IPC 채널, electron preload 새 메서드, main process 다른 핸들러 변경.
  - `git push`, runner process 관리, 새 의존성 추가.

## Notes

- Source order: `tickets/inbox/order_186.md` (`AI 스킬 탭 로딩 지연 + 클릭 시 조회수 누적 문제`).
- 후보 위치 검증:
  - 로딩 지연 — `run_list` (skill-project.sh line ~800) 가 enumerate 후 매 스킬마다 `usage_get_field` 를 4회 호출하고, `usage_get_field` 는 매 호출마다 Python3 + `json.load` 를 새로 띄운다. 80개 스킬 기준 ~320회 Python 부팅 → 탭 진입 시 체감 지연. 단일 Python 인보케이션으로 전체 dump 후 bash 측에서 lookup 만 하면 부트 비용이 1회로 떨어진다.
  - 조회수 — `run_view` (line ~851) 가 `usage_record_event ... view` 를 무조건 호출한다. desktop `controlSkill` (`apps/desktop/src/main.js` line ~3750) 와 renderer `viewSkill` (`main.tsx` line ~1718) 사이에 read-only 옵션이 없어 manual 클릭이 그대로 카운터에 누적된다. AGENTS.md rule 18e 가 정의한 view_count 의미(RAG retrieval 신호) 와 충돌.
- Trade-off: option C (manual_open_count 분리 카운터 신설) 는 sidecar 스키마 마이그레이션이 필요해 본 PRD 범위에서 제외. option B (`--no-bump` CLI 플래그 + 데스크톱 opt-out) 가 가장 좁고 회귀 위험이 낮음.
- 파서 호환: `parseSkillViewOutput` 은 `view_count=` / `last_viewed_at=` 헤더 라인을 그대로 사용하므로 stdout 형식 보존이 필수.
- 회귀 가드: usage.json 의 atomic write (mkstemp + os.replace) 는 view 경로에서 이번에 호출되지 않으므로 직접적인 영향이 없다. list 경로의 read-only 최적화에서도 sidecar 자체에는 쓰지 않는다.
- 후속(별도 PRD 후보, 본 PRD 처리 X): renderer 측 skill list virtualization, `manual_open_count` 분리 카운터 도입, RAG 카운트 증가의 query-당-1회 dedup 검증 telemetry.
