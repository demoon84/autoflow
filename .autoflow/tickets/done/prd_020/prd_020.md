# Project PRD

## Project

- ID: prd_020
- Title: Auto-resume finish-pass when an inprogress ticket has Result: passed but no commit / merge
- AI:
- Status: draft

## Core Scope

- Goal: Impl AI 의 tick 이 (1) 코드 수정 → (2) 검증 통과 → (3) 티켓 markdown 의 `Result: passed by ...` 까지 적은 뒤 (4) `finish-ticket-owner.sh pass` 호출 직전에 죽는 경우, 다음 tick 누구든 자동으로 그 finish 단계만 마무리해 inline merge → done 까지 흐르게 만든다. 현재는 이런 ticket 이 inprogress 에 영원히 남아 사용자가 수동 개입해야 한다.
- In Scope:
  - `runtime/board-scripts/start-ticket-owner.sh` 의 ticket claim 흐름에 새 분기 추가:
    1. 활성 / 보유 / adoptable inprogress 티켓을 결정한 직후, 그 티켓의 `## Result` / `## Verification` 블록을 검사한다.
    2. `Result: passed by ...` 마커 또는 `Verification.Result: passed ...` 가 있고, 동시에 ticket 위치가 inprogress (즉 finish 가 아직 안 돈 상태) 면 "auto-finish" 경로로 진입.
    3. auto-finish 경로:
       - worktree 의 uncommitted 변경이 있으면 `autoflow-recover` 가짜 user 로 단일 commit 생성 (스크립트 안에서 git -C 사용).
       - 이미 commit 이 worktree 에만 있고 main 보다 앞서 있으면 그대로 이어서 사용.
       - 새 worktree commit hash 를 ticket 의 `Worktree Commit` 필드에 채우고 `Integration Status` 를 `awaiting_merge` 로 갱신.
       - `finish-ticket-owner.sh "$id" pass "auto-resumed by recovery path"` 를 inline 으로 invoke (이미 있는 inline merge 흐름 재사용).
    4. auto-finish 가 성공하면 ticket 이 done/ 으로 이동되고 wiki/auto commit 까지 그대로 흐른다.
  - 같은 회복 로직을 `merge-ready-ticket.sh` 의 시작부 (`resolve_ready_ticket_file`) 에서도 재사용 가능하도록 helper 함수 (`recover_passed_inprogress_ticket`) 를 `common.sh` 에 추가.
  - `bin/autoflow doctor` 에 새 체크 `check.passed_inprogress_recovery_pending` 추가: "Result: passed 인데 inprogress/ 에 남은 ticket" 카운트가 0 보다 크면 `warning` 으로 노출.
  - `runtime/board-scripts/start-ticket-owner.sh` 의 새 경로가 호출됐는지 emit 하는 log key 추가 (`source=auto_resumed_finish_pass`) — 데스크톱 / 메트릭 추적용.
- Out of Scope:
  - "Result: passed" 가 아닌 다른 종료 마커 (예: 부분 verify, fail, blocked) 의 자동 회복.
  - Plan AI / Wiki AI 의 동일 회복 (이 두 role 은 long-running ticket 이 아니라 부적합).
  - codex 의 prompt 수정으로 finish 호출 누락 방지 (별도 prompt-tuning PRD 가 필요하면 분리).
  - 합성 commit 의 author 를 사용자 아이덴티티로 두는 정책 (현재는 `autoflow-recover@local.test` 고정).

## Global Rules

- Allowed Paths are relative to the host project root.
- Verification commands run from the host project root.
- 절대 `git push` 하지 않음 — 회복은 local commit + local merge 까지만.
- 회복으로 만들어진 commit 메시지는 `[tickets_NNN] auto-resumed finish pass after dropped tick` 등 명시적 표기.

## Global Acceptance Criteria

- [ ] `tickets/inprogress/tickets_NNN.md` 의 `Result` 가 `passed by ...` 이고 worktree 의 변경이 commit 되지 않은 상태에서 owner runner 의 다음 tick 이 돌면, 자동으로 worktree 를 commit 하고 finish-pass 가 호출되어 ticket 이 done/ 으로 이동한다.
- [ ] 동일 상태에서 worktree 가 이미 commit 만 있고 ticket 만 finish 안 된 경우, 합성 commit 없이 기존 commit 으로 finish-pass 가 호출된다.
- [ ] 회복 후 wiki update + local commit 까지 inline merge 흐름이 그대로 동작한다.
- [ ] `Result: passed` 가 아닌 ticket (`pending`, `failed`, blocked) 은 자동 회복 분기를 타지 않고 기존 흐름 유지.
- [ ] `bin/autoflow doctor` 가 미완료 회복 ticket 이 있을 때 `check.passed_inprogress_recovery_pending=warning` 을 보고하고, 0 일 때 `=ok`.
- [ ] runner 로그에 `source=auto_resumed_finish_pass` 가 발생한 경우 데스크톱의 진행률 카드에서 `last_result` 로 보인다.
- [ ] 기존 정상 ticket 흐름 (Plan AI / Impl AI 가 처음부터 실행하는 케이스) 회귀 없음 — start-ticket-owner.sh 의 dry-run / 일반 claim 출력 형식이 동일.
- [ ] `bash -n` syntax check 통과, `bin/autoflow doctor` status=ok.
