## Core Scope

- Goal: runner 가 실제로 Codex/Claude/Gemini/OpenCode adapter 호출을 수행 중이면 `.autoflow/runners/state/<runner>.state` 의 최근 활동 시각이 계속 갱신되어 desktop "응답 지연" 라벨, `runnerHeartbeatStale`, monitoring 진단이 active 호출을 stuck 으로 잘못 판단하지 않게 한다.
- In Scope:
  - `packages/cli/run-role.sh` 의 adapter 호출 wrapper 에 adapter 실행 중 heartbeat 를 추가한다. 기본 주기는 30초이며, test 에서 줄일 수 있도록 env 로 조정 가능해야 한다.
  - adapter 호출 시작 직후 state 에 `active_stage=adapter_running` 과 최신 `last_event_at` 을 기록하고, 호출이 끝나면 기존 tick finish 경로의 stage/result 기록으로 정리한다.
  - adapter stdout/stderr sidecar 또는 streaming chunk 관찰이 가능한 경우 `last_adapter_chunk_at=<ISO8601>` 를 갱신한다. 구현 방식은 stdout/stderr 파일 mtime/size polling 또는 wrapper-level chunk hook 중 기존 구조에 더 안전한 쪽을 선택한다.
  - state write 는 atomic 하게 수행하고, heartbeat 가 `active_item`, `active_ticket_id`, `active_ticket_title`, `active_spec_ref`, recovery fields, log paths 같은 기존 state 필드를 지우지 않아야 한다.
  - `packages/cli/runners-project.sh` 의 `list_runners` 출력에 `last_adapter_chunk_at` 을 포함해 Desktop 이 사용할 수 있게 한다.
  - `apps/desktop/src/main.js` 는 runner list key=value 출력의 `last_adapter_chunk_at` 을 renderer 모델로 전달한다.
  - `apps/desktop/src/renderer/main.tsx` 의 `runnerHeartbeatStale` 는 `last_event_at` 과 `last_adapter_chunk_at` 중 최신 시각을 기준으로 판단하고, `activeStage === "adapter_running"` 인 runner 가 최근 heartbeat/chunk 를 내고 있으면 "응답 지연" 으로 표시하지 않는다.
  - smoke test 를 추가해 긴 fake adapter 호출 중 state `last_event_at` 이 heartbeat 주기 안에서 갱신되고 `active_stage=adapter_running` 이 관찰되는지 확인한다.
- Out of Scope:
  - `order_159` 의 전체 라벨 단계화 UX(예: 5~15분 의심, 15분+ 경고 색상 체계) 구현.
  - token telemetry stale 복구(`prd_177` / `order_162`)와 metrics 집계 변경.
  - `run_with_timeout` process-tree cleanup 재작성. 해당 계약은 `prd_142` 완료 범위이며 이번 작업은 회귀 검증만 한다.
  - `run_with_timeout` stdin 보존 재작성. 해당 계약은 `prd_125` 완료 범위이며 이번 작업은 회귀 검증만 한다.
  - runner stop/graceful stop 정책 변경(`prd_135`, `prd_167` 계열).
  - runner 또는 OS process 직접 정리, background process cleanup, `git push`.
