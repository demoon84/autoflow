# Project Spec

## Meta

- PRD Key: prd_008
- Title: Auto re-plan rejected tickets in ticket-owner mode (bounded retry loop)
- Status: populated

## Goal

ticket-owner 모드에서 `tickets/reject/` 에 머무는 ticket 이 사람의 수동 개입 없이도 자동 재진입(replan)되어 다시 todo 로 흘러 들어가도록 한다. 무한 재시도를 막기 위해 ticket 단위 retry 카운트와 상한을 둔다. 실패 사유는 새 mini-plan 작성 시 컨텍스트로 보존된다.

## Core Scope

### In Scope

- `runtime/board-scripts/start-ticket-owner.sh` (와 live mirror `.autoflow/scripts/start-ticket-owner.sh`) 의 처리 순서 확장:
  1. 기존: active context → owned inprogress → todo → legacy verifier → backlog spec
  2. 추가: backlog spec 검색 직전에 `find_replannable_reject` 로 reject/ 스캔. 재진입 가능한 reject 가 있으면 `replan_reject_to_todo` 로 todo 로 복원해 즉시 claim.
- `runtime/board-scripts/common.sh` (와 live mirror) 에 헬퍼 함수 추가:
  - `list_reject_tickets_for_replan` — `tickets/reject/reject_*.md` 파일들을 mtime 오름차순으로 출력 (가장 오래된 것부터).
  - `ticket_retry_count` / `bump_ticket_retry_count` — ticket 의 `## Retry` 섹션 또는 `- Retry Count:` 필드 read/write.
  - `find_replannable_reject` — retry 가 상한(`AUTOFLOW_REJECT_MAX_RETRIES`, 기본값 2) 미만인 첫 reject ticket 반환.
  - `replan_reject_to_todo` — reject 파일을 `tickets/todo/tickets_NNN.md` 로 이동, Stage 를 `todo` 로 reset, Owner / Claimed By / Execution Owner / Verifier Owner 를 빈 값으로 초기화, Worktree 섹션을 `pending_claim` 으로 reset, `## Reject History` 섹션에 직전 Reject Reason 보존, retry count +1.
- `## Reject History` 섹션 도입 — 누적 재시도 기록 (timestamp, prior reject reason, prior verifier log path) 을 하나의 ticket 에 누적. 사용자가 한 ticket 의 재시도 변천사를 한 곳에서 본다.
- 새 환경 변수:
  - `AUTOFLOW_REJECT_MAX_RETRIES` (기본 2) — 한 ticket 이 reject → todo 재진입할 수 있는 횟수 상한. 도달 시 더 이상 자동 replan 하지 않고 reject 에 그대로 둔다.
  - `AUTOFLOW_REJECT_AUTO_REPLAN=off` — 전체 비활성화 (수동 모드).
- 출력 보강 (`start-ticket-owner.sh`):
  - replan 이 발생하면 `source=replan` 으로 표기, `reject_origin=tickets/reject/reject_NNN.md`, `retry_count=N` 등 메타 키 추가.
  - 상한 초과로 skip 한 reject 가 있으면 `replan_skipped.<i>=<rel>` / `replan_skipped.<i>.reason=max_retries_reached`.
- ticket-owner agent 계약 갱신:
  - `.autoflow/agents/ticket-owner-agent.md` + `scaffold/board/agents/ticket-owner-agent.md`: Procedure 에 "If start-ticket-owner returns source=replan, treat the prior `## Reject History` entries as constraints when writing the new mini-plan; address the most recent reject reason explicitly" 단계 추가.
- AGENTS.md / CLAUDE.md / README.md 의 Reject 흐름 단락 업데이트 — "ticket-owner 모드도 자동 재계획을 가진다" 명시. 레거시 `#plan` 흐름 설명은 유지(병행 가능).
- 데스크톱 표시 보강 — 작업 흐름의 반려 pin 카드 부제 / layer help text 가 "자동 재시도 N회까지" 또는 "자동 재시도 후 보류" 같은 의미를 반영하도록 한 줄 갱신. (UI 컴포넌트 자체는 변경 없음)
- smoke 테스트 추가 시나리오:
  - 기존 owner smoke 가 그대로 통과 (retry 환경 변수 미설정 시 회귀 없음).
  - 새 단위 검증: reject ticket 1건이 있는 보드에서 `start-ticket-owner.sh` 실행 → status=ok, source=replan, retry_count=1, ticket 이 todo/ 로 이동되어 있음.
  - max retries 도달 케이스: retry_count 가 상한이면 `start-ticket-owner.sh` 가 reject 를 건너뛰고 `replan_skipped.*` 만 출력.

### Out of Scope

- LLM 기반 reject reason 분석 (옵션 C). 본 PRD 는 deterministic retry 만.
- `#plan` 레거시 planner 흐름의 동작 변경.
- reject 사유에 따라 다른 처리 정책(예: 검증 실패 vs 범위 위반) — 모든 reject 를 동일하게 다룬다. 분기는 후속 PRD.
- worktree 강제 재생성 / branch rename 자동화 — replan 시 기존 ticket worktree 필드를 비워 다음 claim 이 알아서 새 worktree 만들도록만.
- reject_NNN 의 verifier_log 와 verify_NNN 파일 정리 — 그대로 logs/ 에 두고, 다음 verification 은 새 verify_NNN 을 inprogress 에 만든다 (overwrite 아님).
- 메트릭 키 변경 (reject_count 그대로). 새 키 `replan_count` 만 추가 가능 (선택).

## Global Rules

- replan 은 deterministic. 같은 reject 에 같은 환경에서 두 번 호출하면 retry_count 만 1 → 2 로 증가하고 동일 결과.
- replan 시 사용자가 손으로 적은 `## Notes` / `## Resume Context` 등 텍스트는 보존. Stage / Owner / Worktree / Verification / Result 같은 머신 상태 필드만 reset.
- `git push` 어떤 자동화에서도 금지 (변경 없음).
- 한국어 UI / 영어 AI 계약 (AGENTS.md rule 15).
- 기본 환경 변수 미설정 시 동작이 켜져 있다 (max_retries=2). 사용자가 `AUTOFLOW_REJECT_AUTO_REPLAN=off` 로 명시하면 비활성.
