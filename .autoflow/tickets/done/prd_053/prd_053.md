- AI: worker-1

## Core Scope

- Goal: Fix the bug visible in the desktop **작업 흐름** page where each runner card's terminal panel (Planner / Worker / 위키봇) shows only one or two static lines — typically a single `timestamp=... event=loop_tick role=... mode=loop exit_code=0 interval_seconds=60` row — instead of a live, character-by-character typing stream of the runner's recent adapter conversation. The 위키봇 card additionally surfaces only a log file path (`rs/logs/wiki-1_2026-04-29T12-39-03Z_stderr.log`) rather than that log's actual tail. After this PRD, every runner card's terminal panel must (a) reveal new content with a visible typing animation rather than instantaneous text swaps, (b) include more than just the latest single `loop_tick` line whenever the runtime/live/stdout/stderr logs have richer content available, and (c) gracefully render the heartbeat tick output when no adapter is configured so the panel is never blank or stuck on a stale single line.
- In Scope:
  - Diagnose (don't blindly patch): trace the data path that feeds the runner card terminal panel.
    - `runnerConversationText(runner)` at [`apps/desktop/src/renderer/main.tsx:2516`](apps/desktop/src/renderer/main.tsx:2516) returns `runner.conversationPreview || runner.lastLogLine`. Check what each falls back to when both adapter output and runtime tick are present, and whether `conversationPreview` is being clobbered by `lastLogLine` due to falsy-string-trim logic.
    - `runnerConversationPreview()` in [`apps/desktop/src/main.js:933`](apps/desktop/src/main.js:933) reads the tail of `lastStdoutLog`, `lastRuntimeLog`, `runnerLiveLogPaths(...)`, and `lastStderrLog` and runs them through `extractAgentConversation()`. Verify (a) the candidate order picks the right log when only the live log has content, (b) `extractAgentConversation` does not over-filter and end up returning a single path-shaped string (the 위키봇 symptom suggests it returned an stderr line carrying the live log filename instead of its body), (c) when all candidates yield empty or non-conversational text, the function falls back to a runtime-tick summary so the renderer always has something to type.
    - `ConversationStream` at [`apps/desktop/src/renderer/main.tsx:2631`](apps/desktop/src/renderer/main.tsx:2631) advances `displayedLength` via `setTimeout` ticks of 16ms with up to 1.5s catch-up. When the input `text` is short (~one line) the visible animation finishes in a single tick and looks static. Adjust the initial display length policy so a fresh stream with previously-empty cache types from 0 (or from a small offset) instead of jumping straight to full length.
  - Fix the bug surface(s) identified above. The expected diagnosis is one or more of: wrong fallback chain in `runnerConversationText`, broken extraction in `extractAgentConversation` (especially for the wiki/coordinator runners that emit stderr file-path lines), and/or stale poll cadence that never refreshes `conversationPreview` for ticks where no adapter ran.
  - Ensure the terminal panel honours the existing `TYPING_TAIL_CHARS` (400) cap and `TYPING_TICK_MS` (16ms) / `TYPING_TARGET_CATCHUP_MS` (1500) constants — only NEW deltas type in character by character; existing cached prefix is rendered immediately so a long historical log does not multi-second-drip on every navigation.
  - Token-frugality: do NOT add a polling loop that re-reads the full runtime log each tick. Reuse the existing read-tail helpers (`readTailText`) and the existing IPC refresh cadence; if the data path is broken, fix the extraction or the fallback, not the polling rate.
  - Update [`.autoflow/wiki/features/desktop-runner-controls.md`](.autoflow/wiki/features/desktop-runner-controls.md) (or add a new `desktop-runner-terminal-panel.md` feature page) describing the fixed contract.
- Out of Scope:
  - Adding new live-log infrastructure on the runner side (the existing `_live_stdout.log` / `_live_stderr.log` files already exist).
  - Reworking the typing-effect easing curve or sound — only the visibility of the animation needs fixing.
  - Changing the IPC refresh cadence between main and renderer.
  - Any wiki / CLI / runner-side change. This PRD is desktop UI only.

## Notes

- 토큰 절약 관점: 본 PRD 는 어댑터 호출이나 런타임 폴링 빈도를 늘리지 않는다. 기존 `readTailText` (8 KB tail) 와 IPC refresh 만으로 표시되는 데이터를 정확히 보여주는 데 초점이 있다. typing 애니메이션 자체가 추가 IPC 를 발생시키지는 않는다.
- 위키봇 카드의 `rs/logs/...stderr.log` 한 줄은 stderr 첫 줄이 그 경로 자체를 적은 것일 가능성이 높음 (러너가 stderr 첫 라인에 자기 로그 위치를 echo). `extractAgentConversation` 가 그 한 줄만 골라낸 것으로 추정. 추출 로직이 conversation 시그니처를 못 찾으면 path-only 라인은 버리고 runtime-tick fallback 으로 가도록 수정한다.
- Risk: 타이핑 애니메이션을 항상 0 부터 시작하게 만들면 사용자가 처음 화면에 들어왔을 때 히스토리 전체가 1.5 초간 드립한다. 그래서 `TYPING_TAIL_CHARS=400` 안에서만 새 델타가 타이핑되도록 보존한다. 캐시 된 prefix 는 즉시 출력.
- Risk: live log 가 partial flush 라 마지막 라인이 잘려 보일 수 있다. 그 경우 typewriter 가 잘린 라인 끝에서 멈춰 버리는 것처럼 보인다. fallback 으로 마지막 newline 까지만 잘라서 보여주는 보정을 검토한다.
- Heartbeat: planner-1 이 본 PRD 를 다음 1 분 tick 안에 backlog → todo 로 승격하면 owner-1 이 worktree 에서 진단/수정/검증을 진행한다. 데스크톱 dev 앱은 이미 `npm run dev` 로 띄워져 있으므로 worker 가 hot reload 로 변경 효과를 즉시 확인할 수 있다.
