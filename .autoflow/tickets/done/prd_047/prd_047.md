# Project PRD

## Project

- ID: prd_047
- Name: Cap verification run output for successful ticket-owner checks
- Title: Reduce successful verification stdout and stderr stored in ticket-owner run files
- Goal: Reduce recurring adapter prompt token usage by compacting successful ticket-owner verification output while preserving enough failure output for debugging.
- AI: Codex
- Status: draft

## Core Scope

- Goal: Change ticket-owner verification run-file output capture so successful checks store a small first/last excerpt with an explicit truncation marker, while failed checks keep a larger diagnostic tail.
- In Scope:
  - Update `.autoflow/scripts/verify-ticket-owner.sh` so pass results compact `## Output` for stdout and stderr to first 20 lines plus last 20 lines when the stream is longer than the pass cap.
  - Keep an explicit truncation marker that records omitted line count for each compacted stream.
  - Lower the default pass output budget from the current 200-line per-stream tail to about 50 lines or less per stream.
  - Preserve a larger failure diagnostic window, defaulting to the existing `AUTOFLOW_VERIFY_OUTPUT_LINES:-200` behavior unless the implementation introduces a clearly named failure-specific override.
  - Apply the same behavior to `runtime/board-scripts/verify-ticket-owner.sh` so source runtime and current board runtime stay aligned.
- Out of Scope:
  - Do not edit `finish-ticket-owner.sh`; it is relevant because it reads the pending run file back into finish context, but the durable size reduction belongs in the run file written by `verify-ticket-owner.sh`.
  - Do not change verification command execution, status/result semantics, ticket claim flow, finish finalization, or wiki updates.
  - Do not remove stdout or stderr sections entirely.
  - Do not change product behavior outside the ticket-owner verification runtime scripts.

## Global Acceptance Criteria

- [ ] Successful verification run files write stdout and stderr as compact first/last excerpts with an explicit truncation marker when a stream exceeds the pass cap.
- [ ] The default successful output budget is no more than 50 lines per stream and does not store the current 200-line tail for each stream.
- [ ] Failed verification run files keep a larger diagnostic tail, preserving the current 200-line default unless an equivalent failure-specific default is introduced.
- [ ] `.autoflow/scripts/verify-ticket-owner.sh` and `runtime/board-scripts/verify-ticket-owner.sh` implement the same output-capping behavior.
- [ ] The run file still contains `### stdout` and `### stderr` sections for both pass and fail outcomes.
- [ ] `finish-ticket-owner.sh` can continue reading the pending run file without any format change outside the smaller `## Output` content.
