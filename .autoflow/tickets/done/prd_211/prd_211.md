# PRD 211 — Planner 의 backlog PRD 우선 처리 + priority 우선 정책

## Goal

Planner tick 에서 `tickets/backlog/prd_*.md` 에 promote 가능한 PRD 가 있고 동시에 `tickets/inbox/order_*.md` 에 신규 quick order 가 있을 때, 같은 priority 안에서는 backlog PRD → todo 변환을 새 PRD 작성보다 먼저 처리한다. priority 가 다르면 priority enum (`critical` > `high` > `normal` > `low`) 이 카테고리(backlog vs inbox)보다 우선한다. 결과적으로 진행 중 작업이 신규 작업보다 먼저 todo 로 빠지면서, 우선순위 역전은 일어나지 않는다.

## Background

- order_189 사용자 요청: "플래너가 order/prd 를 처리할 때 prd 가 남아 있다면 prd 를 먼저 작성하게 하는 건 어떨까". 의도는 **이미 backlog 에 있는 PRD 를 todo 로 흘려보내는 것** 을 새 PRD 작성보다 우선하자는 것.
- 현재 `.autoflow/scripts/start-plan.sh` dispatcher 순서:
  1. Branch 1: express order → todo
  2. Branch 1.9: order-generated backlog PRD → todo (`select_order_generated_spec`)
  3. Branch 2: inbox order → AI generated PRD (`select_inbox_order`)
  4. Branch 3: backlog PRD → todo (`select_populated_spec`)
- 즉 일반 backlog PRD 의 todo 변환(Branch 3) 이 inbox order 처리(Branch 2) 보다 **뒤에** 있어 backlog 가 채워진 채로 inbox 만 계속 빠지는 패턴이 가능. order_189 는 이 순서를 바꾸자는 제안.
- AGENTS.md rule 20 의 priority enum (`critical` / `high` / `normal` / `low`) 은 그대로 유지하고, **카테고리 간 priority 비교** 만 추가한다.

## Decision

후보 A 를 채택한다 (order_189 제안 그대로):

- 기본 정책: 같은 priority 안에서 `backlog PRD → todo` 가 `inbox order → 새 PRD` 보다 우선.
- priority 가 다르면 priority enum 이 카테고리보다 우선. 즉:
  - `high` inbox order > `normal` backlog PRD (inbox 먼저)
  - `normal` backlog PRD > `normal` inbox order (backlog 먼저)
- starvation 가드: 같은 backlog PRD 가 `AUTOFLOW_BACKLOG_FIRST_STUCK_LIMIT` (기본 3) tick 연속 같은 위치에서 promote 못 하면 그 tick 에 한해 inbox 처리로 fallback. 가드 카운터는 `.autoflow/runners/state/backlog-first-stuck.json` 에 PRD 파일명 기준 누적, promote 성공 시 0으로 reset.
- retry order (`order_*_retry_*.md`) 는 정책에서 제외 — 기존 PRD 와 묶여 있으므로 inbox 카테고리지만 backlog PRD 와 자연스럽게 묶어 처리한다 (현재 동작 유지).
- express order (rule 21) 는 PRD 단계를 건너뛰므로 정책 영향 없음 (Branch 1 그대로 최우선).
- `select_order_generated_spec` (Branch 1.9) 도 그대로 최우선 유지 — 자기 order 에서 생성한 PRD 를 같은 tick 에 todo 로 흘리는 것은 starvation 방지에 유리.

## Scope

- `.autoflow/scripts/start-plan.sh` 의 dispatcher 분기 순서/조건 변경.
  - Branch 1 (express), Branch 1.9 (order-generated backlog PRD) 는 그대로.
  - Branch 2 (inbox order) 와 Branch 3 (backlog PRD) 사이에 priority + starvation 가드 기반 선택 로직을 둔다. 두 후보를 모두 본 뒤 선택한다.
  - inbox order 가 retry order 인 경우는 새 정책 적용 대상에서 빼고, 기존처럼 일반 inbox 흐름으로 처리한다.
- 가드 상태 파일 `.autoflow/runners/state/backlog-first-stuck.json` 의 read/write 헬퍼.
  - 누락 / 깨진 JSON 은 `{}` 로 회복 (1원칙 보존).
- `.autoflow/agents/plan-to-ticket-agent.md` 의 Procedure 절에 새 정책을 한 줄로 명시.
- `AGENTS.md` rule 20 본문에 카테고리 간 priority 비교 규칙을 한 줄 보강 (rule 번호 신설 없이 rule 20 안에 추가).
- backlog/inbox 후보가 모두 priority normal 이고 가드가 안 걸린 일반 경우의 단일 path 단위 테스트는 별도 PRD 가 아닌 본 PRD 의 verification 으로 처리 (수동 시나리오 + script smoke).

## Out of Scope

- priority enum 자체 변경 (기존 `critical` / `high` / `normal` / `low` 유지).
- worker / wiki runner 의 dispatcher (이번 PRD 는 planner 만).
- retry order 의 routing rule 자체 (현재 동작 유지, 정책에서 단순 제외).
- queue helper 의 priority sort 코드 (이미 존재 — 재사용만).
- `tickets/check/` 같은 폐기된 ledger 와 관련된 옛 흐름 (영향 없음 확인만).

- order_189 본문 보존: 사용자가 트레이드오프(우선순위 역전, starvation, retry order 예외, express 예외) 를 직접 짚어 줬다. 본 PRD 는 그 의견을 그대로 반영한다.
