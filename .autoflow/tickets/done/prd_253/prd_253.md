## Goal

PTY 모드 runner의 progress slider가 항상 "대기"에 머무는 회귀 문제를 근본적으로 해결한다.

현재의 filesystem inference 방식(`enrichRunnerActiveTicketFromFs`)은 다른 worker의 `main.js` 편집과 충돌할 때마다 회귀한다. 대신 LLM 자신이 작업 단계 전환 시점에 `runner-stage.js`를 직접 호출하여 state 파일을 능동적으로 갱신하도록 한다. Explicit signal이므로 race/cache/회귀에서 자유롭다.

## Background

- PTY 모드에선 `active_stage` / `active_ticket_id` state 필드를 갱신하는 주체가 없음
- `enrichRunnerActiveTicketFromFs` 같은 filesystem 추론 enrichment는 main.js 편집 충돌 시 회귀 반복
- LLM이 각 phase 전환 시 명시적으로 state를 쓰면 UI fs.watch가 자동으로 새로고침 (별도 IPC 추가 불필요)

## Scope

1. `.autoflow/scripts/runner-stage.js` 신규 스크립트 작성
2. `ticket-owner-agent.md`, `plan-to-ticket-agent.md`, `wiki-maintainer-agent.md`에 호출 의무 명시
3. `apps/desktop/src/main.js`의 `buildInitialPrompt` worker 섹션에 `runner-stage.js` 호출 단계 추가

## Acceptance Criteria (Global)

- `runner-stage.js <stage> [--runner <id>] [--ticket <ticket-id>] [--note <text>]` 인터페이스로 동작
- state 파일(`runners/state/<runner>.state`)의 `active_stage`, `active_ticket_id`, `active_ticket_title`, `last_event_at` 갱신
- JSONL audit log `runners/logs/<runner>-stage.log` 기록
- agent.md 3개에 단계별 호출 의무 명시
- buildInitialPrompt worker 섹션에 runner-stage.js 호출 문구 추가
- PTY / legacy 모드 모두 호환
- 실패 시 exit 0 + stderr 경고만 (LLM 메인 흐름 차단 금지)

## Notes

- `enrichRunnerActiveTicketFromFs` 같은 기존 filesystem inference는 fallback으로 유지 (LLM이 호출하지 않은 경우 대비) — 이 PRD에서 제거하지 않음
- runner-stage.js는 PROJECT_ROOT 기준으로 동작해야 worker/wiki 양쪽에서 동일 경로 사용 가능
- LLM이 매 단계 호출 안 하더라도 claim 직후 1회만 호출해도 inprogress 진입 보장
- state 파일에 `active_stage` 필드가 없으면 추가, 있으면 덮어씀 (upsert)
