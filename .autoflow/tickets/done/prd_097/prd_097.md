# Project PRD

## Meta

- Project Key: prd_097
- Title: Reject auto-replan 기본 retry 한도 3회로 변경
- Status: populated

## Goal

Reject auto-replan 의 기본 retry 한도(`AUTOFLOW_REJECT_MAX_RETRIES` 미지정 시 사용하는 기본값)를 기존 10회에서 3회로 낮춘다. 동일한 reject 티켓을 무한히 다시 todo 로 되돌리는 비용을 줄이고, 더 빠르게 `needs_user` 복구 분기로 진입시키는 것이 목표다. 환경변수 `AUTOFLOW_REJECT_MAX_RETRIES` 명시값과 ticket 별 `Retry → Max Retries` override 동작은 그대로 유지하고, 보드 runtime parser 계약, key=value 출력, runner id, role 키, smoke test 의 기존 시나리오 의미는 회귀 없이 보존한다.

## Core Scope

### In Scope

- `.autoflow/scripts/common.sh` 의 `reject_max_retries()` 안에서 `AUTOFLOW_REJECT_MAX_RETRIES` 미지정 시의 fallback 기본값을 `10` → `3` 으로 변경한다. 같은 함수의 두 번째 fallback (`raw` 가 비숫자/빈 문자열일 때 사용하는 `"2"` 안전망) 은 별도 의미라 본 PRD 에서는 변경하지 않는다.
- `runtime/board-scripts/common.sh` 의 동일 위치 `reject_max_retries()` fallback 기본값을 `10` → `3` 으로 동일하게 맞춰, live 보드와 runtime template 미러 사이의 기본값 표류를 막는다.
- 위 두 파일 변경 후 `bash -n` 로 문법 회귀가 없음을 확인하고, 기존 `tests/smoke/ticket-owner-replan-smoke.sh` 가 끝까지 통과하는지 (특히 `Max Retries: 2` ticket-level override 시나리오와 `replan_skipped.*` 키 출력) 회귀 확인한다.
- 필요 최소 한도에서, 위 변경 의미를 사람이 읽는 보드 운영 문서에 반영한다. `scaffold/board/AGENTS.md`, `scaffold/board/agents/plan-to-ticket-agent.md`, `scaffold/board/automations/README.md` 가 reject auto-replan 한도를 언급하는 문장에 "(기본 3회)" 같은 짧은 보조 문구를 덧붙이거나, 기존 표현이 새 기본값과 충돌하지 않으면 그대로 둔다. parser 계약이 있는 키/필드/명령은 손대지 않는다.

### Out of Scope

- ticket 별 `Retry → Max Retries` override 동작 변경, `extract_scalar_field_in_section` 호출 경로 변경.
- `AUTOFLOW_REJECT_AUTO_REPLAN` on/off 정책 변경, reject 파일 라이프사이클(reject/ ↔ todo/) 흐름 변경.
- `start-plan.sh`, `start-ticket-owner.sh`, `verify-ticket-owner.sh`, `finish-ticket-owner.sh`, `merge-ready-ticket.sh` 의 분기 변경.
- `replan_skipped.*` 출력 키 이름 변경, key=value 포맷 변경, `recovery_state` / `failure_class` 의미 변경.
- 새 환경변수 추가, 새 CLI 옵션 추가, 새 ticket 필드 추가.
- `dogfood-board/scripts/common.sh` 같이 본 memo Allowed Paths 에 포함되지 않은 추가 mirror 의 갱신.
- `.autoflow/agents/*`, `protocols/*`, `reference/*`, README, AGENTS.md, CLAUDE.md 본문 수정.
- 기존 done 티켓/PRD/wiki 의 회고 문구를 새 기본값으로 소급 수정.
- 새로운 smoke test 추가 또는 기존 smoke test 의 시나리오 확장.

## Notes

- 출처 memo: `tickets/inbox/memo_062.md` ("Reject limit 3회로 변경"). Plan AI 는 memo 의 Hint(Scope/Allowed Paths/Verification) 와 저장소의 현재 코드를 결합해, 기본값 상수 한 줄을 두 미러에서 동기 변경하는 가장 좁은 해석으로 정리했다.
- 현재 코드 위치 확인: `reject_max_retries()` 는 `.autoflow/scripts/common.sh:1022-1031` 과 `runtime/board-scripts/common.sh:1022-1031` 두 곳에 동일하게 존재하며, `${AUTOFLOW_REJECT_MAX_RETRIES:-10}` 로 기본값을 노출한다. 같은 함수의 두 번째 fallback (`if [ -z "$raw" ]; then raw="2"; fi`) 은 환경변수가 비숫자일 때의 안전망이라 별도 의미를 가지므로 본 PRD 에서는 손대지 않는다.
- 보드의 reject 큐 현황 (start-plan 출력): `replan_skipped.1=tickets/reject/reject_003.md` (retry_count=10), `replan_skipped.2=tickets/reject/reject_071.md` (retry_count=8), `replan_skipped.3=tickets/reject/reject_074.md` (retry_count=1). 모두 이미 `recovery_state=needs_user` 로 주차되어 있어, 기본값을 3회로 낮춰도 과거 reject 의 상태를 다시 자동 재시도로 풀어내는 회귀가 발생하지 않는다 (retry_count 가 새 기본값 이상인 경우 그대로 needs_user 유지).
- smoke 영향 분석: `tests/smoke/ticket-owner-replan-smoke.sh` 는 retry-limit 시나리오에서 ticket-level `Max Retries: 2` override 를 명시적으로 주입하므로(라인 ~203-208), 환경변수 기본값 10→3 변경은 기존 단언(`replan_skipped.1.retry_count=2`, `failure_class=retry_limit`, `recovery_state=needs_user`) 에 영향을 주지 않는다. 첫 시나리오의 단일 replan(`retry_count=1` → 다음 시도가 한 번 더 가능) 도 새 기본값 3 아래라 통과한다. 따라서 smoke 는 그대로 유지하면 충분하다.
- scaffold 문서들은 현재 한도를 숫자 없이 `AUTOFLOW_REJECT_MAX_RETRIES` 심볼로만 참조한다. 새 기본값을 명시적으로 전달하고 싶다면 같은 문장에 "(기본 3회)" 같은 짧은 한국어 보조구를 덧붙이는 정도로 충분하다. parser 가 읽는 키/필드/경로는 변경하지 않는다.
- 호환성 / 운영 영향: 외부에서 `AUTOFLOW_REJECT_MAX_RETRIES` 를 명시한 사용자는 영향을 받지 않는다. 미지정 사용자만 한도가 10→3 으로 줄어들어, 같은 reject 티켓이 평균 3회 자동 재시도 후 빠르게 `needs_user` 로 진입한다. 기존에 이미 `replan_skipped.*` 로 주차된 티켓들은 변화 없이 reject/ 에 남아 사용자/오케스트레이터의 재계획 결정을 기다린다.
- Plan AI wiki 검색 결과: `AUTOFLOW_REJECT_MAX_RETRIES` 검색은 `prd_092` (planner 오케스트레이터에서 needs_user 전이 다룸), `prd_008` (초기 reject auto-replan 도입, 당시 기본값 2), `prd_092/Todo-090.md`, `prd_008/Todo-008.md` 와 매칭된다. `reject auto-replan` 검색은 `prd_008` 의 초기 도입 이력과 wiki 의 project overview/log/index, planner-worker-lifecycle 답변 노트를 가리킨다. 본 PRD 는 이 기존 결정과 호환된다 — 기본값만 운영 경험에 맞춰 더 보수적인 3 으로 재조정하는 것이며, retry 메커니즘/recovery 분기/parser 계약은 prd_008 / prd_092 가 정한 그대로다.
- 실행 시 권장 mini-plan: ① 두 미러 (`reject_max_retries()`) 동시 수정, ② `bash -n` 통과 확인, ③ scaffold 문서 표현 정렬(필요 시), ④ smoke 실행. smoke 는 임시 디렉토리에서 git init/autoflow init 까지 수행하므로 first-run 비용이 있을 수 있다.
