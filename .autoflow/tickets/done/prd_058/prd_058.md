## Core Scope

- Goal: runtime display helper 와 desktop renderer 의 runner/worker label normalization 을 확장해, 같은 역할의 enabled runner 가 단일 인스턴스일 때만 사용자 노출 라벨에서 숫자 접미사를 생략한다.
- In Scope:
  - `.autoflow/scripts/common.sh` 와 `runtime/board-scripts/common.sh` 의 `display_worker_id()` 계열 helper 가 `owner-N` / `worker-N` / `ai-N` 을 `worker-N` 으로 정규화하되, 현재 enabled runner topology 에서 ticket-owner 역할이 1개뿐이면 사용자 노출 출력은 `worker` 로 표시하도록 확장한다.
  - `planner-1` 과 `wiki-1` 도 같은 원칙으로 사용자 노출 출력에서 단일 인스턴스이면 `planner`, `위키봇` 으로 표시한다.
  - 같은 역할 runner 가 2개 이상 enabled 되는 경우에는 `planner-1`, `worker-1`, `worker-2`, `위키봇-1`처럼 suffix 를 다시 표시해 구분 가능성을 유지한다.
  - Desktop renderer 의 runner 카드, ticket/Notes/log preview, workflow board metadata 에서 storage id 를 그대로 노출하는 경로를 찾아 동일한 display helper 또는 equivalent mapping 을 적용한다.
  - CLI runner command/summary 출력에서 사용자에게 보여 주는 runner id label 이 있다면 같은 display policy 를 적용한다.
  - live board scripts, source runtime scripts, dogfood board script copy 가 같은 display behavior 를 유지하도록 필요한 mirror update 를 수행한다.
  - policy docs 가 user-visible worker label 을 항상 `worker-N` 으로만 요구한다면 단일 runner 예외를 명시한다.
- Out of Scope:
  - `planner-1`, `owner-1`, `wiki-1` storage id, `.autoflow/runners/config.toml`, runner state filenames, runtime role keys, branch names, ticket ids 는 변경하지 않는다.
  - ticket parser fields such as `Claimed By`, `Execution AI`, `Verifier AI`, `Owner`, `Worktree`, `Project Key` 의 machine-readable compatibility 를 깨지 않는다.
  - historical `tickets/done/`, verification records, logs 를 대량 수정하지 않는다.
  - wiki derived pages 를 직접 수정하지 않는다. 완료 후 wiki-maintainer 가 파생 지식을 갱신한다.
  - runner topology, heartbeat behavior, claim/finish semantics, provider/model/reasoning settings 은 변경하지 않는다.
