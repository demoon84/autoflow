# Project PRD

## Goal

`/order`, `#order`, `$order` 로 호출되는 quick-intake 흐름의 사용자 노출 wording 을 전부 "order" 기준으로 정렬한다. 대상은 (1) Codex/Claude order 스킬 4 종, (2) `bin/autoflow` 의 사용법/예시 출력과 dispatch case, (3) `packages/cli/memo-project.sh` 가 inbox 파일에 채워 넣는 본문/Source/Planner Contract 문구와 stderr 메시지/`next_action` 안내, (4) `scaffold/board/README.md`, `scaffold/board/AGENTS.md`, `scaffold/board/reference/memo.md`, `scaffold/board/agents/plan-to-ticket-agent.md`, `scaffold/board/automations/README.md` 의 사용자/AI 가 읽는 설명, (5) `scaffold/host/AGENTS.md`, `scaffold/host/CLAUDE.md`, repo 루트 `README.md`, `CLAUDE.md`, `packages/cli/README.md` 의 host 가이드, (6) `apps/desktop/src/renderer/main.tsx` 의 사용자 노출 한글 안내문이다. 동시에 `tickets/inbox/memo_NNN.md` 파일 prefix 자체와 보드 스캐너 (`scripts/start-plan.sh`)는 본 PRD 의 Allowed Paths 밖이라 그대로 유지하고, 호환 진입을 위해 `bin/autoflow` 의 dispatch case `memo)` 와 `packages/cli/memo-project.sh` 자체 파일명/내부 var 이름은 file-pattern 호환 근거로 보존한다.

## Core Scope

### In Scope

- `packages/cli/memo-project.sh` 를 정리한다.
  - 생성되는 inbox markdown 의 사용자 노출 본문을 `# Autoflow Order`, `## Order`, `- ID: order_NNN`, `- Source: autoflow order create` 로 바꾼다 (concrete 예: `- ID: order_063`).
  - `## Planner Contract` 안의 "this memo", "memo intake" 같은 사용자/AI 노출 문구를 "this order", "order intake" 로 바꾼다.
  - stderr 메시지 (`Memo source file not found`, `Memo request is required ...`) 를 `Order source file not found`, `Order request is required ...` 로 바꾼다.
  - 종료 후 stdout 의 `next_action` 안내 문구도 order 기준 한국어/영문으로 정렬한다 (`promote this memo` → `promote this order`).
  - parser/runtime 호환 키인 `status=created`, `status=blocked`, `reason=memo_already_exists`, `memo_id=...`, `memo_file=...`, `project_root=...`, `board_root=...`, `board_dir_name=...` key 이름과 value format, 그리고 inbox 파일 prefix `memo_NNN.md` 자체는 변경하지 않는다 (보드 스캐너 호환 근거).
  - 내부 함수명 `normalize_memo_id`/`next_memo_id` 와 내부 변수명 `memo_id`/`memo_file`/`memo_dir` 도 file-pattern (`memo_NNN.md`) 호환 근거로 유지한다 (사용자 노출 표면이 아니며 변경 시 출력 키와 어긋남 위험).
- `bin/autoflow` usage/help 와 dispatch.
  - `usage()` 의 명령어 라인 `autoflow memo create [...]` 와 Examples 라인 `autoflow memo create ...` 를 모두 `autoflow order create [...]` 로 바꾼다. README 표 등에서도 동일.
  - `case "$cmd"` 에 `order)` case 를 추가하고 동일하게 `packages/cli/memo-project.sh` 로 dispatch 한다. 기존 `memo)` case 는 호환 alias 로 그대로 유지한다 (사용자 도움말에서는 더 이상 노출하지 않는다 — 보드 스캐너/스크립트가 내부적으로 호출할 수 있어 dispatch 자체는 보존).
- `.codex/skills/order/SKILL.md`, `.claude/skills/order/SKILL.md`, `integrations/codex/skills/order/SKILL.md`, `integrations/claude/skills/order/SKILL.md` 4 파일을 정리한다.
  - frontmatter `description`, 본문 제목 (`# Autoflow Quick Order`), 동작 단계 텍스트, "memo" 호환 안내문(`Renamed from memo; ... unchanged`)을 사용자 노출 표면에서 모두 제거한다.
  - 단계 표현을 "order 노트를 `tickets/inbox/` 에 저장한다", "save the order under `tickets/inbox/`" 식으로 정렬한다. 파일 prefix 가 현재 `memo_` 인 점은 사용자 노출 wording 에서는 다루지 않는다 (스캐너/CLI 가 내부에서 처리).
  - CLI 호출 예시는 모두 `autoflow order create ...` 로 바꾼다.
  - "tell the user the memo path" → "tell the user the saved order file path" 식으로 평이한 한국어/영문으로 정렬.
- `scaffold/board/README.md`, `scaffold/board/AGENTS.md`, `scaffold/board/reference/memo.md`, `scaffold/board/agents/plan-to-ticket-agent.md`, `scaffold/board/automations/README.md` 안에서 사용자/AI 가 읽는 설명을 정리한다.
  - 섹션 제목 (`## Memo Intake Rules`), 본문 (`Quick memo intake`, `quick memo`, `memo intake`, `the memo`, `memo path`, `Plan AI promotes clear memos ...`) 를 order 기준으로 정렬한다.
  - 사용자 노출 명령 호출은 모두 `autoflow order create` 로 정렬.
  - 파일 prefix 자체에 대한 사용자 노출 설명에서 literal `memo_NNN.md` 표현은 제거하거나 `tickets/inbox/` 경로만 노출. parser/agent 계약상 필요한 곳은 concrete 예 `memo_001.md` 로 대체하거나 "현재 보드 스캐너가 인식하는 inbox 파일 prefix" 로 우회 표현 (literal 문자열 `memo_NNN` 이 본문에 남지 않게 한다).
  - `tickets/done/<project-key>/memo_NNN.md` 같은 archive 경로 설명도 동일 원칙으로 우회.
  - Plan AI / AI 계약 문장에서 "this memo" 같은 wording 은 "this order" 로 정렬, 파일 흐름 자체는 그대로 (planner 가 inbox 의 quick order 노트를 generated PRD/todo 로 promote 한다는 의미).
- `scaffold/host/AGENTS.md`, `scaffold/host/CLAUDE.md` 의 host 가이드와 trigger 섹션 (`#order`, `/order`, `$order`) 본문을 정리한다.
  - "이전 이름 `#memo` 에서 변경됨" 류 호환 안내문, "파일 이름 prefix `memo_` 와 CLI `autoflow memo create` 는 호환을 위해 그대로 둠" 류 문구, literal `memo_NNN.md` 경로 표기를 사용자 노출 표면에서 제거한다.
  - 사용자 노출 명령 호출은 모두 `autoflow order create` 로 정렬.
  - host 가이드의 quick-intake 흐름 설명을 "order 노트는 `tickets/inbox/` 에 저장되고 Plan AI 가 generated PRD/todo 로 승격한다" 로 평이하게 정리.
- repo 루트 `README.md`, `CLAUDE.md`, `packages/cli/README.md` 안의 사용자 노출 wording 을 동일 원칙으로 정렬한다. CLI 명령 표/Examples 에서 `autoflow memo create` 표기를 모두 `autoflow order create` 로 바꾼다. `memo create` 한 단어 표기도 `order create` 로 정렬. `tickets/inbox/memo_NNN.md` literal 경로 표기는 위와 동일하게 우회.
- `apps/desktop/src/renderer/main.tsx` 의 사용자 노출 한국어 안내문/aria 라벨/툴팁 등에서 "memo" 표현을 "order" 로 정렬한다 (예: `emptyDescription="tickets/inbox/memo_*.md 또는 tickets/done의 memo가 생기면 폴더 기준 칸반으로 표시됩니다."` 같은 줄). 사용자 노출 표면이 아닌 내부 type/kind/status key (`"memo"` 문자열, `isMemoBoardFile`, `isInboxMemoBoardFile`, `/^memo_\d+\.md$/i` 정규식, `deleteInboxMemoFile` 호출) 은 file-pattern 호환과 preload API 호환 근거로 그대로 유지한다.

- 호환성 / 운영 영향: 사용자/AI 가 본 가이드와 스킬 안내가 모두 "order" 로 일관된다. 기존 보드/스캐너/runtime 는 변경되지 않아 기존 inbox 파일과 done memo 파일의 의미가 유지된다. 데스크톱 UI 의 사용자 노출 라벨도 "Order" 로 일관된다.
- 실행 시 권장 mini-plan: ① CLI 생성기 (`memo-project.sh`) 본문/메시지부터 정렬해 사용자 노출 양성 패턴 (`# Autoflow Order`, `## Order`, `- ID: order_NNN`, `autoflow order create`) 을 한 곳에 확보, ② `bin/autoflow` usage/Examples 에 `autoflow order create` 노출 + `order)` dispatch case 추가, ③ 4 개 order 스킬 정리, ④ scaffold/board 5 파일 + scaffold/host 2 파일 + repo 루트 README/CLAUDE + `packages/cli/README.md` 정리, ⑤ `apps/desktop/src/renderer/main.tsx` 의 사용자 노출 한국어 안내문 정렬, ⑥ verification 명령 (rg positive + ! rg negative + bash -n + tsc + check-syntax) 일괄 실행해 회귀/누락 확인.
