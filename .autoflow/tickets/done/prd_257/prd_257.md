## Goal

세션 로그 파싱 기반의 token 추론 대신 LLM이 매 tick 종료 시점에 `runner-tokens.js`를 직접 호출해 자기 측정값을 능동 보고하는 방식으로 통일한다.

CLI(claude/codex/gemini)별 세션 로그 포맷은 버전마다 변동돼 fragile하다. LLM은 자기 turn의 token usage를 정확히 알고 있으므로, 직접 보고하면 더 정확하고 일관적인 신호를 얻을 수 있다. `runner-stage.js`와 같은 push 패턴으로 일관성을 유지한다.

## Background

- 세션 로그 파싱(PRD_251 token watcher)은 CLI 버전에 따라 포맷이 달라 회귀 위험
- LLM은 TUI status bar에 turn token usage를 표시 — 자기가 보고하는 값이 가장 정확
- runner-stage.js(PRD 253)와 동일한 패턴으로 push > pull 일관성 확보
- 세션 로그 파싱은 fallback으로만 유지 (`token_source=session_log`)

## Scope

1. `.autoflow/scripts/runner-tokens.js` 신규 스크립트 작성
2. `ticket-owner-agent.md`, `plan-to-ticket-agent.md`, `wiki-maintainer-agent.md`에 매 turn 종료 시 호출 의무 명시
3. `apps/desktop/src/main.js` buildInitialPrompt에 호출 문구 추가
4. `apps/desktop/src/renderer/main.tsx` UI에서 cumulative/last_turn tokens 표시

## Acceptance Criteria (Global)

- `runner-tokens.js report` 인터페이스로 state 파일의 token 필드 7개 갱신
- JSONL audit log `runners/logs/<runner>-tokens.log` 기록
- agent.md 3개에 매 turn 종료 시 호출 의무 명시
- buildInitialPrompt에 호출 문구 추가
- UI 카드 하단 "0 tokens" 영역에 cumulative/last_turn 실시간 표시
- session log watcher fallback 유지 (`token_source=session_log`)
- 실패 시 exit 0 + stderr 경고만 (LLM 메인 흐름 차단 금지)
- tick-id로 중복 보고 방지

## Notes

- runner-stage.js(PRD 253)와 동일한 공통 state file writer 유틸 공유 설계 권장
- tick-id는 LLM이 생성한 UUID 또는 순서 카운터 — 같은 tick 두 번 보고 방지용
- session log watcher(PRD_251 token watcher)는 fallback으로 유지 — 이 스크립트가 primary
- UI에서 `token_source` 값으로 primary(llm_reported)/fallback(session_log) 구분 표시 가능
