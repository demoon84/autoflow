# Project PRD

## Project

- ID: prd_016
- Title: Pin AI progress board to a 2-left / 1-right tall layout when three runners are present
- AI:
- Status: draft

## Core Scope

- Goal: 작업 흐름 페이지의 AI 진행 보드가 현재 `repeat(2, 1fr)` 그리드로 채워지면서 3-runner 토폴로지에서 우하단이 항상 비어 있는 문제를 해결한다. Impl AI 카드를 오른쪽 칼럼 전체 높이로 띄우고, Plan AI 와 Wiki AI 가 왼쪽 칼럼에 위/아래로 쌓이도록 해 빈 공간을 제거한다. Impl AI 가 출력량이 가장 많기 때문에 더 큰 영역에 배치되는 것이 가독성 측면에서도 자연스럽다.
- In Scope:
  - `apps/desktop/src/renderer/main.tsx` 의 `<div className="ai-progress-board">` 에 runner role 정보를 셀에 전달할 수 있도록 `data-runner-role={runner.role}` 속성과, 보드 컨테이너에 `data-runner-count={runners.length}` 속성을 추가.
  - `apps/desktop/src/renderer/styles.css` 의 `.ai-progress-board` 에 다음 분기 추가:
    - `data-runner-count="3"` 일 때:
      - `grid-template-columns: minmax(0, 1fr) minmax(0, 1fr)`
      - `grid-template-rows: minmax(220px, 1fr) minmax(220px, 1fr)`
      - `grid-template-areas: "plan impl" "wiki impl"` (Impl AI 가 우측 두 행 span)
      - 각 셀: `[data-runner-role="planner"|"plan"] { grid-area: plan }`, `[data-runner-role="ticket-owner"|"owner"] { grid-area: impl }`, `[data-runner-role="wiki-maintainer"|"wiki"] { grid-area: wiki }`
    - 기타 count (1, 2, 4+): 현재 `repeat(2, 1fr)` 동작 유지.
  - 좁은 viewport (예: < 720px) 에서는 grid-template-areas 를 단일 칼럼 stack 으로 fallback 해 raster 깨짐 방지.
- Out of Scope:
  - 4 이상 runner 가 있을 때의 새 레이아웃 정의 (현재 자동 흐름 유지).
  - AI 카드 내부 (헤더, 단계 표시기, 콘텐츠 박스) 의 디자인 변경.
  - Plan AI ↔ Wiki AI 의 위/아래 순서 사용자 토글.
  - 다른 페이지(`티켓 정보`, `통계`, `Wiki`)의 그리드.

## Global Acceptance Criteria

- [ ] 데스크톱에 정확히 3개 runner (planner / ticket-owner / wiki-maintainer) 가 있을 때, 작업 흐름 페이지의 AI 보드가 좌측 2단 (위 Plan AI, 아래 Wiki AI) + 우측 1단 (Impl AI 가 두 행 높이 span) 으로 보인다.
- [ ] 같은 레이아웃에서 우하단이 빈 공간 없이 채워진다 (현재 스크린샷의 빈 사각형이 제거된다).
- [ ] runner 수가 3 이 아닐 때 (1, 2, 4+) 는 현재 `repeat(2, 1fr)` 자동 흐름이 그대로 동작하고 시각 회귀 없음.
- [ ] viewport width < 720px 에서는 세 카드가 단일 칼럼으로 세로 적층되어 잘리지 않는다.
- [ ] `cd apps/desktop && npx tsc --noEmit` 가 0 errors.
- [ ] `cd apps/desktop && npm run check` 가 통과한다.
- [ ] 시각 회귀: `티켓 정보`, `AI 관리`, `Wiki`, `통계` 등 다른 사이드 페이지가 영향 없음.

## Notes

- role 슬러그가 `planner-1` → `planner` / `owner-1` → `worker` / `wiki-1` → `wiki-maintainer` 로 바뀌는 prd_012 가 동일 영역을 건드리므로, 본 티켓을 그 다음에 처리하면 머지 충돌이 가장 적다 (Impl AI 가 ID 만 바꾸고, 본 티켓은 layout 만 건드림).
- `data-runner-role` 은 정확히 `runner.role` 문자열을 그대로 노출 — 추후 다른 role 이 추가될 때도 자동 매핑 가능.
- 4+ runner 가 등장하면 별도 PRD 로 그 케이스의 레이아웃 정의 (현재 자동 흐름은 visual 회귀 없음).
