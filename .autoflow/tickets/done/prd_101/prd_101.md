# Project PRD

## Meta

- Project Key: prd_101
- Title: Wiki query 호출처에 RAG chunk retrieval 디폴트 적용
- Status: populated

## Goal

planner/worker/wiki-maintainer 가 prior-decision 컨텍스트를 끌어올 때 호출하는 `autoflow wiki query` 명령이 file-level grep 대신 chunk-level RAG retrieval(`--rag`)을 디폴트로 쓰도록 agent 가이드 안내문과 runtime 안내 문구를 정렬한다. 코어 RAG 구현(`packages/cli/wiki-project.sh` 의 `wiki_query_score_chunks`, `tests/smoke/wiki-rag-query-smoke.sh`)은 prd_099 단계에서 이미 들어와 있고 본 PRD 는 호출처(가이드 문구) 만 정렬한다. CLI 디폴트, 출력 포맷, smoke 계약, 임베딩 기반 RAG 도입은 본 PRD 범위가 아니다.

## Core Scope

### In Scope

- `.autoflow/agents/plan-to-ticket-agent.md` 와 그 scaffold 짝(`scaffold/board/agents/plan-to-ticket-agent.md`) 의 wiki query 호출 안내문에 `--rag` 를 추가한다.
  - 대상 줄: `Available commands` 절의 `autoflow wiki query --term <text> — surfaces prior decisions/learnings ...` 예시.
  - 절차 본문(`5. If source=memo-inbox ... run autoflow wiki query`, `6. Before drafting a new plan, run autoflow wiki query ...`) 의 한국어/영문 안내도 동일 패턴으로 정렬.
- `.autoflow/agents/ticket-owner-agent.md` 와 `scaffold/board/agents/ticket-owner-agent.md` 의 안내문에 `--rag` 를 추가한다.
  - 대상 줄: `Available commands` 절의 `autoflow wiki query --term <text>` 예시, 그리고 본문 `4. Run autoflow wiki query with 1–3 distinctive terms ...` 절차.
- `.autoflow/agents/wiki-maintainer-agent.md` 와 `scaffold/board/agents/wiki-maintainer-agent.md` 의 안내문 5 곳을 정렬한다.
  - 헤더 가이드 (`when it is set, for example "$AUTOFLOW_CLI" wiki query --term <text>`).
  - `Available commands` 절의 `wiki query --term <text>` 베이스라인 예시.
  - 같은 절의 `wiki query --synth`, `wiki query --synth --save-as <slug>` 호출 예시 (RAG 와 synth 는 독립 옵션이므로 `--rag --synth` / `--rag --synth --save-as <slug>` 로 함께 표기).
  - 본문 `7. ... run autoflow wiki query --term <text> instead of grepping by hand` 절.
- `runtime/board-scripts/run-role.sh:1535` 의 사용자 노출 안내문(`Run a wiki context pass before planning or implementation: use 'autoflow wiki query' with distinctive terms ...`) 을 `'autoflow wiki query --rag'` 로 정렬한다.
- 위 변경에 맞춰, RAG 디폴트 권장의 근거(`chunk_start_line`/`chunk_end_line` 메타로 어느 영역을 봐야 하는지까지 알려주고, 큰 wiki 파일에서 prompt budget 을 절감한다는 정성적 이유)를 각 가이드의 짧은 한 문장으로만 첨가한다. 길이가 늘어나는 본문 추가는 하지 않는다.
- 검증 명령은 사용자 노출 표면에서 `wiki query --term ... --rag` 패턴이 존재하고, RAG 없이 동일 호출을 권장하는 누락분이 없음을 rg 로 확인한다.

### Out of Scope

- `tests/smoke/wiki-rag-query-smoke.sh` 의 어떤 변경도 본 PRD 범위 밖.
- `AUTOFLOW_WIKI_RAG_CHUNK_LINES` / `AUTOFLOW_WIKI_RAG_CHUNK_OVERLAP` 디폴트 변경.
- 임베딩/벡터 DB 기반의 의미적 RAG 도입.

- `.autoflow/scripts/` 의 신규 mirror 추가(현재 `.autoflow/scripts/run-role.sh` 가 존재하지 않으므로 신규 생성 금지).
- 데스크톱 UI, runner state, ticket lifecycle, 보드 스캐너, smoke 계약 변경.
- 기존 `tickets/done/` 안의 ticket 본문에 남은 `wiki query --term ...` 인용문 소급 수정.

## Notes

- 코어 RAG 구현은 prd_099 의 wiki rag chunk retrieval 추가/병합 commit(`be54198`, `178ad79`)에서 이미 들어와 있다(`packages/cli/wiki-project.sh` 의 `wiki_query_rag_chunk_lines`, `wiki_query_rag_chunk_overlap`, `wiki_query_score_chunks` + smoke `tests/smoke/wiki-rag-query-smoke.sh`). 본 PRD 는 그 위에 호출처 디폴트만 정렬한다.
- 정량 근거: 60 파일/2,386 라인 wiki 에서 동일 query(`--term "RAG" --term "chunk"`) 의 file-mode payload(예: `wiki/log.md` 11,959B 전체) 가 RAG-mode 에서 32 라인 chunk(약 70% 절감) 로 줄어든다. 작은 결정/feature 페이지(40~50 라인) 는 chunk 1~2 개로 거의 동일하지만 `chunk_start_line`/`chunk_end_line` 메타로 위치 정보가 추가된다.
- `--rag` 와 `--synth` 는 독립 옵션이다(`packages/cli/wiki-project.sh` 의 `run_query` argument 11번째). 함께 쓰면 chunk 단위 retrieval 위에 synth 를 얹는다. wiki-maintainer 가이드의 synth 호출 예시도 `--rag --synth` 로 정렬해 동일 chunk 메타가 synth 입력에 포함되게 한다.
- 호환성 영향: CLI 디폴트는 그대로 file-mode 이므로 본 PRD 외부에서 `autoflow wiki query --term ...` 를 호출하는 자동화/스크립트(`.autoflow/scripts/`, `tests/smoke/*.sh`, 기타 runtime) 는 영향이 없다. `--rag` 를 명시한 새 경로만 chunk 출력을 받는다.
- 역회귀 방지: 가이드 양쪽(`.autoflow/agents/` 와 `scaffold/board/agents/`) 동시 갱신을 acceptance 로 강제한다. 한쪽만 바뀌면 `init`/`upgrade` 시 scaffold 가 live board 로 덮어써 drift 가 다시 발생한다.
- 실행 mini-plan 권장: ① wiki-maintainer 가이드(`.autoflow` + `scaffold`) 5 곳 정렬, ② plan-to-ticket 가이드 2 곳 정렬, ③ ticket-owner 가이드 2 곳 정렬, ④ `runtime/board-scripts/run-role.sh:1535` 안내 문자열 정렬, ⑤ rg positive/negative + diff drift + bash -n 일괄 실행해 회귀/누락 확인.
