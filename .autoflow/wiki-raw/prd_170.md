## Core Scope

- Goal: `tickets/inbox/order_150.md` 가 보고한 1원칙 위반 패턴을 runtime 레벨에서 복구한다. `needs_user` / 오래된 `repairing` ticket 이 `inprogress` 에 무한 잔류해 `worker.state.last_result=ticket_stage_blocked` 와 결합되는 현상을 막는다.
- In Scope:
  - `.autoflow/scripts/start-plan.sh` 와 `runtime/board-scripts/start-plan.sh` 에 planner tick 시작부 recovery scan 을 추가해 `Stage: blocked` + `Recovery State.Status: needs_user` ticket 을 worker claim 대상에서 분리된 parked 상태로 처리한다. 구현은 기존 board 구조와 호환되도록 `tickets/reject/` archive 또는 명시적 parked/needs_user stage 중 repo 패턴에 맞는 가장 좁은 방식으로 선택한다.
  - `Recovery State.Status: repairing` 이 설정된 ticket 이 설정 가능한 timeout(기본 30분)을 넘으면 planner 가 `needs_user` 또는 retry 가능한 recovery action 으로 escalate 하고, Evidence / Planner Decision / Owner Resume Instruction 을 갱신한다.
  - `packages/cli/run-role.sh` 의 worker(`ticket`) tick 진입부가 active item 의 `Recovery State.Status` 를 확인해 `needs_user` 또는 timeout 지난 `repairing` 이면 stale active item / `ticket_stage_blocked` 표시를 해제하고 다음 todo claim 이 가능하도록 한다.
  - sidecar(`.autoflow/scripts/*`) 와 install template(`runtime/board-scripts/*`) 동작을 대칭으로 유지한다.
  - board 변경은 idempotent 하게 남긴다. 같은 evidence / decision 이면 Notes 를 중복 append 하지 않는다.
- Out of Scope:
  - prd_168 의 check ledger 자기참조 live-lock fixpoint guard.
  - prd_169 의 cleanup 이후 stale `worker.state.last_result=ticket_stage_blocked` 단독 reset. 단, active item recovery status 확인과 직접 연결되는 reset 은 포함한다.
  - ticket 구현 코드 작성, 실제 runner / OS process kill, worktree 삭제, `git push`.
  - desktop UI 표시 로직 변경. state 와 board stage 가 정확해지면 UI 는 자연 회복해야 한다.
