## Core Scope

- Goal: token budget 정책이 실제 최신 호출 데이터를 근거로만 `token_budget_exceeded`를 기록하게 만들고, stale token-cache 또는 갱신 중단 telemetry 상태에서는 worker/ticket 진행을 차단하지 않도록 fail-safe 경로를 추가한다.
- In Scope:
  - `packages/cli/run-role.sh`와 `runtime/board-scripts/run-role.sh`에서 `token_budget_exceeded`가 기록되는 정확한 trigger와 budget source를 확인하고, stale 데이터로 판정되는 경로를 차단한다.
  - token budget 판정이 `.autoflow/metrics/token-cache.tsv` 같은 legacy/cache 파일을 참조한다면 마지막 row timestamp 또는 파일 mtime 기준 freshness 검사를 추가한다.
  - freshness 한도는 `AUTOFLOW_TOKEN_BUDGET_MAX_DATA_AGE_SECONDS`로 조정 가능하게 하고 기본값은 3600초로 둔다.
  - 데이터가 stale이면 `token_budget_exceeded` 대신 budget 검사를 skip 했다는 경고성 state/log marker를 남기고 adapter/ticket 진행 자체는 실패로 분류하지 않는다.
  - `.autoflow/telemetry/runs.jsonl`에 최신 호출 token 정보가 있으면 그 데이터를 우선한다. legacy token-cache fallback이 fresh telemetry를 덮어쓰지 않게 한다.
  - runner state/list 또는 desktop bridge가 `last_result=token_budget_exceeded`를 표시하는 경우 stale skip 결과를 구분해 사용자에게 false failure처럼 보이지 않게 한다.
  - smoke test를 추가해 stale token-cache와 최신 telemetry fixture가 함께 있을 때 budget 판정이 최신 telemetry 또는 skip marker를 사용하고 stale cache만으로 exceeded를 기록하지 않는지 확인한다.
- Out of Scope:
  - PRD_177의 token telemetry record/finalize 전체 복구 재작성.
  - `tickets_177` / PRD_178의 adapter-running heartbeat 구현 변경.
  - token budget 임계값 자체의 정책 재설계나 비용 환산 UI 추가.
  - 과거 telemetry/token-cache 데이터 삭제 또는 backfill.
  - runner 또는 OS process 직접 정리, `git push`.
