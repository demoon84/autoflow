## Core Scope

- Goal: `tickets/inbox/order_153.md`가 보고한 `readBoard` 내부 `Promise.all` fan-out 약점을 제거한다. `status`, runner list, `doctor`, `metrics`, `stop-hook-status`, `watch-status` 중 하나가 reject, spawn error, timeout, hang 상태가 되어도 나머지 board snapshot과 fallback metadata로 renderer가 계속 그릴 수 있어야 한다.
- In Scope:
  - `apps/desktop/src/main.js`의 `readBoard` 내부 6개 진단 호출 fan-out을 `Promise.allSettled` 또는 동등한 safe wrapper로 바꿔 단일 실패가 전체 handler reject로 전파되지 않게 한다.
  - `runAutoflowCachedOrRefresh` / `listRunnersCachedOrRefresh`의 background refresh가 개별 timeout(예: 15000ms)과 bounded child cleanup을 사용하도록 한다. timeout 결과는 `ok: false`, `cancelled: true`, `stderr` 경고, `readBoardFallback` metadata를 가진 fallback result로 관찰 가능해야 한다.
  - `readBoard` 호출의 `withTimeout(handler, 30000)` 안전망은 유지하되, 내부 subcall timeout이 더 짧아 정상 운영에서는 handler-level 30000ms timeout이 발생하지 않게 한다.
  - `readBoardMeta.fallbackSources` 또는 동등한 machine-readable metadata에 timeout / rejected / fallback source가 남아 renderer와 로그에서 부분 실패를 구분할 수 있게 한다.
  - `autoflow:listRunners`, `autoflow:listRunnerArtifacts`, `autoflow:readBoardFile`, `autoflow:getConfig`는 같은 unbounded child timeout 문제가 있는지 코드 검토하고, 이번 ticket 안에서 바꾸는 경우 `apps/desktop/src/main.js` 안의 같은 timeout helper만 재사용한다.
- Out of Scope:
  - `prd_140`에서 이미 완료된 readBoard stale-while-revalidate / cache fallback 구조 전면 재작성.
  - `prd_144`에서 완료된 standalone `autoflow:listRunners` fork-bomb guard 재작업.
  - `prd_104`의 metrics cold-start critical-path 분리 재작업.
  - renderer UI redesign, preload IPC contract 변경, runner CLI 성능 최적화, board/wiki/runtime 자동화 정책 변경.
  - planner/worker/adapter runner process를 planner가 직접 관리하거나 `git push`하는 작업.

## Global Acceptance Criteria

- [ ] `apps/desktop/src/main.js`의 `readBoard` 내부 6개 진단 호출은 `Promise.allSettled` 또는 동등한 safe wrapper를 사용해 단일 rejected promise가 전체 `autoflow:readBoard` IPC reject로 이어지지 않는다.
- [ ] `status`, runner list, `doctor`, `metrics`, `stop-hook-status`, `watch-status` 각각은 개별 timeout(기본 15000ms 안팎) 또는 공유 helper를 통해 handler-level 30000ms보다 짧게 fallback result를 반환한다.
- [ ] timeout 또는 reject가 난 diagnostic result는 `ok: false` 또는 `partial/fallback` 계열 field, `cancelled`/`signal`/`stderr` 증거, source 이름을 포함해 `readBoardMeta.fallbackSources`에서 식별된다.
- [ ] 한 diagnostic refresh가 timeout되어도 나머지 성공/캐시/파일 목록 결과는 board snapshot에 유지되고, renderer는 빈 전체 board 대신 부분 board를 받을 수 있다.

- [ ] standalone `autoflow:listRunners` IPC의 TTL/inflight guard와 timeout cleanup은 `prd_144`의 완료 범위를 유지하며, 이번 변경이 그 경로를 다시 direct spawn 구조로 되돌리지 않는다.
